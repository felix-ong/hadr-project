HADR Monitor

What this codebase is, what to build first, and the ten questions to settle before starting.

1What this codebase is

This is a starter template with no application code in it yet — sixteen files, all Markdown and workflow YAML. It frames a three-day exercise: by Wednesday afternoon the repository must contain an agent that watches three live disaster feeds (GDACS, USGS earthquakes, ReliefWeb), filters and assesses events, and publishes a morning situation report to dashboard.html at 08:30 Singapore time, unattended, staying quiet when nothing has changed. The README is explicit that how is deliberately unspecified — that is the course.

What the template does contain is a working method, encoded in its scaffolding:

2What I would build first

The thinnest end-to-end pipe: USGS → normalised snapshot → change check → dashboard. One feed, no model call, no dedup, no assessment — but every stage of the eventual system exists and runs.

  1. Fetch and normalise USGS (all_day.geojson) into a small, stable event schema on disk — id, magnitude, place, time, updated, alert. USGS first because it needs no approval, has no ambiguity about identity within itself, and regenerates every minute, so it is the easiest feed to test against and the fastest to fail against.
  2. Write the change-detection script in scripts/: compare the new snapshot to the last, exit 0 on change and 1 on no-change (or vice-versa — whatever the workflow branches on). This script is the keystone: the disabled sitrep workflow cannot be enabled without it, and its existence forces the "deterministic first, model second" shape from day one.
  3. Render a minimal dashboard.html from the snapshot — a dated list of events above a threshold. Ugly is fine; published is the point.
Why this order: the risky unknowns in this project are not the feeds — they are the seams (scheduling, change detection, quiet mornings, publishing). A vertical slice puts every seam under test on day one, then GDACS, ReliefWeb, cross-feed identity, and model-written assessments are added into a pipeline that already runs. It also matches the repo's own slice template: done in two minutes of a reviewer's time, everything else explicitly out of scope.

3Ten questions before starting

  1. Who reads the 08:30 report, and what decision does it drive? Everything downstream — severity threshold, geographic scope, tone, length — depends on whether the reader is a duty officer deciding on activation or a team wanting ambient awareness.
  2. What is the noise floor? USGS ships ~200 events a day, mostly tiny; GDACS marks most events Green. Does a Green M4.6 in Japan appear at all, one line, or nowhere? Is the filter magnitude, alert colour, population affected, or region?
  3. Is scope global, or Asia-Pacific weighted? The 08:30 Singapore anchor suggests a regional audience. A global report and a regional one filter and rank very differently.
  4. What exactly counts as "changed" — and what does "stays quiet" mean? New event only, or also an alert-colour upgrade or magnitude revision? And on a quiet morning: no publish at all, or a dated "nothing to report" line so readers can tell silence from failure?
  5. Which GDACS alert field is authoritative, and do we correct after a colour change? Events carry alertlevel, alertscore, episodealertlevel and episodealertscore, and a colour can change after we have reported. Report-then-upgrade needs a policy, not an accident.
  6. What makes two records the same event across feeds? The same earthquake arrives from USGS and GDACS (both via NEIC) under different ids, and from ReliefWeb days later with a GLIDE number. Is GLIDE the join key when present, with time-and-distance proximity as fallback? Is a duplicate line worse than a wrongly merged one?
  7. What happens to an already-published report when the event changes underneath it? USGS revises magnitudes and occasionally deletes events outright. Do we amend the published dashboard, append a correction, or only ever report forward?
  8. Has the ReliefWeb appname been requested, and is RSS-only acceptable meanwhile? API approval takes time we may not have this week. If the answer is RSS for now, the design should isolate ReliefWeb behind an interface so the API can be swapped in without touching the pipeline.
  9. Where is dashboard.html served, and does history matter? Committed to the repo, GitHub Pages, or elsewhere? And is each morning's report archived (a sitrep trail has real HADR value) or overwritten in place?
  10. What goes in CLAUDE.md, and what budget do the headless runs have? Language, test command, and conventions are blank and the README says to fill them before the first prompt. Also: which model for the sitrep call, what monthly cost is acceptable for a daily scheduled run plus polling, and are the required secrets already set in Actions?
